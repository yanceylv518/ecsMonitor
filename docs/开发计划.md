# 阿里云 ECS 监控桌面程序 — 开发计划

> 配套文档：[设计方案.md](./设计方案.md)
> 路线：**UI 先行** —— 先搭代码骨架，用 Mock 数据源把前端完整做出来，再接真实阿里云 API 替换数据源

## 核心思路：数据源抽象

在骨架阶段就定义统一的数据源接口，Mock 与真实实现可互换，UI 全程只面向接口开发：

```python
class MetricSource(ABC):
    def list_instances(self) -> list[InstanceInfo]: ...
    def fetch_metrics(self, instance_ids, metric_names,
                      start_ms, end_ms) -> list[DataPoint]: ...

class MockMetricSource(MetricSource):   # 阶段 1：生成仿真数据
class AliyunMetricSource(MetricSource): # 阶段 3：真实 API
```

- Mock 生成**形态逼真**的数据：CPU 带日间波峰的正弦+噪声、内存缓慢爬升、网络突发尖峰，并模拟 3~5 台虚构实例（含一台「停止」态、一台「CPU 告警」态），保证 UI 的各种状态都能开发和演示
- Mock 数据同样经 `storage.py` 写入 SQLite —— 存储层、查询层从第一天起就是真实的，后期只换数据源
- `--mock` 启动参数**永久保留**：后续演示、UI 回归、无网环境开发都用它，不是用完即弃的临时代码

## 开发阶段总览

| 阶段 | 内容 | 产出物 | 预估工作量 |
|:--:|------|--------|:--:|
| 0 | 环境 + 完整代码骨架 + 数据源接口 | 可运行的空壳程序 | 1 天 |
| 1 | Mock 数据源 + 存储层 + 全部 UI | 用假数据完整跑起来的桌面程序 | 4~5 天 |
| 2 | 真实采集层（阿里云 API） | CLI 跑通真实采集入库 | 2 天 |
| 3 | 接入真实数据源联调 | 真实数据驱动的完整程序 | 1 天 |
| 4 | 托盘与打磨 | 托盘常驻、告警、日志、自启 | 1~2 天 |
| 5 | 测试与打包 | Windows 安装包 | 1~2 天 |

总计约 **10~13 个工作日**。

## 阶段 1：Mock 数据源 + 全部 UI（4~5 天）✅ 已完成 2026-07-09（tag: v0.2-mock-ui）

> 全程 `--mock` 模式开发，不需要阿里云账号。

**1a. 数据底座（约 1 天）**

- [ ] `core/storage.py`：建表（`metrics`、`instances` + 唯一索引）、批量 `INSERT OR IGNORE`、范围查询、最新值查询、过期清理 —— 按设计方案 5.2 实现，这层就是最终实现，不是 mock
- [ ] `core/mock_source.py`：3~5 台虚构实例（含停止态、告警态各一台）；各指标仿真曲线（CPU 日间波峰正弦+噪声、内存缓升、网络尖峰）；支持按任意时间范围生成历史数据，保证详情页 7d 视图有数据
- [ ] `core/config.py`：配置数据类 + JSON 读写 + 默认值
- [ ] `worker.py`：`CollectorWorker(QThread)` 循环「从 MetricSource 拉数据 → 入库 → 发信号 → 可中断 sleep」；信号：`data_updated`、`instances_updated`、`collect_error`、`state_changed`；支持暂停/恢复/立即采集/配置热更新/优雅退出 —— worker 面向接口，此时驱动的是 mock，阶段 3 换真实源时**零改动**
- [ ] 单元测试：storage 全覆盖（内存 SQLite）、mock_source 出数正确性

**1b. UI 开发（约 3~4 天）**

- [ ] `ui/main_window.py`：侧边导航 + QStackedWidget 三页 + 状态栏（最近采集时间/数据点数/错误提示）
- [ ] `ui/overview_page.py`：
  - [ ] 实例卡片（名称/ID/状态/CPU%/内存%/公网出带宽最新值），流式网格布局
  - [ ] `data_updated` 信号触发刷新；CPU/内存超阈值标红；停止态灰显
  - [ ] 双击卡片 → 跳详情页并选中该实例
- [ ] `ui/detail_page.py`：
  - [ ] 实例下拉 + 时间范围切换（1h / 6h / 24h / 7d）
  - [ ] pyqtgraph 分组子图：CPU+内存（%）、网络（bps）、磁盘（Bps），共享 X 轴，本地时区
  - [ ] 十字光标 + 悬浮数值；新数据增量刷新；大范围数据降采样
- [ ] `ui/settings_page.py`：完整表单（凭证、地域、实例选择、指标勾选、间隔/保留期/阈值），保存写 config.json 并热更新 worker；「测试连接」按钮先接 mock（必成功），阶段 3 接真实验证
- [ ] 空态处理：无实例、无数据时的占位提示

**验收标准**

1. `python main.py --mock` 启动后：总览页显示虚构实例卡片并每周期刷新，告警实例标红、停止实例灰显
2. 详情页四个时间范围曲线完整、切换流畅、悬浮取值正确
3. 设置页改采集间隔后立即生效（无需重启）
4. `pytest` 全绿

## 开发约定

- **分支/提交**：单人开发 main 直推，每阶段完成打 tag（`v0.1-skeleton`、`v0.2-mock-ui`…）
- **测试边界**：core 层强制单测（API mock、内存 SQLite）；UI 层以手动验收清单为主
- **密钥纪律**：真实 AccessKey 不进代码、不进 git、不进日志；开发用环境变量，测试用 RAM 只读子账号
- **Mock 纪律**：`--mock` 是长期能力，mock 代码与真实代码同等质量要求；禁止在 UI 层写「if mock」分支 —— 差异全部收敛在 MetricSource 实现里
- **跨平台注意**：开发在 Linux、目标在 Windows，keyring 后端、路径、注册表相关代码留平台分支，阶段 5 前在 Windows 完整跑一次
